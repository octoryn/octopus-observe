# 安全策略

## 报告漏洞

安全漏洞请**不要提交公开 issue**。

请通过 GitHub Security Advisories(仓库 Security 页的 "Report a vulnerability")
或邮件 **security@octoryn.com** 私下报告。请包含描述、复现步骤与影响。我们力争在
数个工作日内确认。

## 范围说明

Observe 是一个**可信事实入口层 (trusted-fact entry layer)**:把不可信的外部事件
转化为规范、不可变的观测。按设计,以下几处与安全相关:

- **不可信边界。** `ingest` 接受任意输入,面对恶意输入**绝不崩溃** —— 校验是唯一
  会拒绝的阶段,每个结果都是返回值(`accepted` / `duplicate` / `rejected` /
  `skipped`),而非抛异常。这由一套对抗性模糊测试覆盖(畸形信封、非有限数值、
  `__proto__` 等恶意键、深层嵌套、unicode)。若发现任何导致抛异常、崩溃或原型
  污染的输入,请报告。
- **防篡改与密钥。** 审计轨迹是一条哈希链,每条观测带有 `integrity` 哈希。
  **不加密钥时,它们是可*察觉*篡改,而非可*防止*篡改** —— 哈希是公开的,拥有存储
  写权限的攻击者可以伪造出内部自洽的记录。要获得真正的抵抗力,请提供
  `auditSecret` / `integritySecret`(带密钥的 HMAC),并用同一密钥校验;像对待任何
  签名密钥一样保护并轮换它。轮换密钥只会为之后的哈希重新奠基 —— 不会回溯性地
  重新校验旧记录。
- **原始事件归档是明文面。** 可选的 `RawEventArchive` 会逐字保存原始输入,可能
  包含 PII/PHI 或密钥。请按合规要求施加静态加密、访问控制与留存窗口
  (`pruneBefore`)。它默认关闭。
- **无网络出站。** Observe 不做任何出站 I/O,且零运行时依赖。把事件送进来、把
  审计/观测送出去,是运维方(与连接器)的职责 —— 保护已存储的数据同理。
